You should take that feedback seriously, but do not overreact and remove BigQuery. BigQuery is the right analytics store for this project; the expensive part is repeatedly scanning raw event data from Grafana and logical views.

Your current dashboard showed about 15M events over two days. At that scale, the design must shift from:

Grafana → logical views → normalized raw audit events

to:

GCS raw audit logs
      ↓
BigQuery normalized hot data
      ↓
Scheduled curated governance tables + aggregate tables
      ↓
Grafana

With on-demand BigQuery, cost is driven mainly by bytes scanned per query, not row count. Logical views do not store results; their underlying query is executed each time the view is queried. That means each Grafana refresh and each panel can rescan underlying data unless you pre-aggregate.

Your sample has several strong governance categories.

The most important decision is this:

That immediately cuts the data used by the dashboard.

Cost-optimized target design
Layer 1 — GCS remains the raw source of truth
GitHub Enterprise Audit Logs
        ↓
GCS .json.log.gz files

Keep raw logs in GCS according to your approved retention policy.

Do not use BigQuery as the only long-term raw archive. GCS is already storing compressed audit logs efficiently and is the right evidence/archive layer.

Keep your existing table:

But treat it as a hot operational table, not the table Grafana reads for every panel.

Normalized detailed events: 30–90 days

Raw GCS audit files:        existing approved retention

github_governance_daily_metrics

Then Grafana runs tiny queries like:

SELECT
  metric_time AS time,
  SUM(event_count) AS events
FROM `github_governance_metrics_15m`
WHERE $__timeFilter(metric_time)
GROUP BY metric_time
ORDER BY metric_time;

That is far cheaper than re-reading millions of audit rows for every dashboard panel.

How to generate the aggregate tables

Use scheduled queries every 15 minutes.

Raw normalized events
        ↓
Scheduled query every 15 min
        ↓
Governance events table
        ↓
Scheduled aggregation
        ↓
15-minute / daily dashboard tables
        ↓
Grafana

Scheduled queries are appropriate here because they let you write flexible GoogleSQL and persist results into destination tables on a recurring schedule.

For your case, I would use scheduled queries rather than materialized views initially.

Why:

Governance classification will have CASE rules and multiple action patterns.
You need controlled backfill and rule-version tracking.
You may need to reclassify historical events when rules change.
Grafana should query simple aggregates, not complex classification SQL.

Materialized views can reduce repeated compute for compatible pre-aggregations, but they have design limitations. They are worth considering later for simple metrics once query patterns stabilize.

Immediate cost controls to implement
1. Require partition filters

Your normalized table is already partitioned by event_time.

Add:

require_partition_filter = true

to the normalized events table.

Add the same protection to:

github_governance_events
github_governance_metrics_15m
github_governance_daily_metrics
github_audit_pipeline_health

This prevents accidental full-table scans. BigQuery can reject queries that do not include a usable partition filter when this setting is enabled.

Do this only after confirming every dashboard query includes one of:

WHERE $__timeFilter(event_time)

or:

WHERE $__timeFilter(started_at)
2. Stop using SELECT *

Never expose raw_event_json through dashboard views.

BigQuery is columnar, so selecting only the required fields avoids scanning unrelated columns. The biggest risk is a dashboard/view query using SELECT * against the normalized event table.

3. Keep Grafana refresh conservative

Your current refresh is:

5 minutes

That is reasonable for Pipeline Health.

For Governance Overview, use:

15 minutes

Do not use 30-second or 1-minute dashboard refreshes against BigQuery. Each panel can execute a separate query, and a dashboard opened by several users multiplies that cost.

4. Set dashboard defaults to short time ranges

Recommended defaults:

Pipeline Health:      Last 24 hours
Governance Overview:  Last 7 days
Executive Summary:    Last 30 days

A user can still choose 90 days, but the default should not scan 90 days every time someone opens Grafana.

5. Separate operational and management dashboards
Dashboard 1: Audit Pipeline Health
- 24-hour default
- 5-minute refresh
- reads health aggregates

Dashboard 2: Governance Overview
- 7-day default
- 15-minute refresh
- reads governance aggregate tables

Dashboard 3: Monthly Governance Report
- 30/90-day range
- refresh off
- reads daily aggregates only

This stops manager dashboards from accidentally behaving like real-time analytics systems.

6. Measure actual spend before optimizing blindly

Create a lightweight daily cost check from BigQuery job metadata.

Use the region where your BigQuery dataset/jobs run. Do not assume it is the same as the Cloud Function region.

SELECT
  DATE(creation_time) AS query_date,
  user_email,
  COUNT(*) AS query_jobs,
  ROUND(SUM(total_bytes_billed) / POW(1024, 3), 3) AS tebibytes_billed
FROM `YOUR_PROJECT_ID.region-YOUR_BIGQUERY_LOCATION.INFORMATION_SCHEMA.JOBS_BY_PROJECT`
WHERE creation_time >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)
  AND job_type = 'QUERY'
  AND statement_type = 'SELECT'
GROUP BY query_date, user_email
ORDER BY query_date DESC, tebibytes_billed DESC;

total_bytes_billed is available in the BigQuery jobs metadata for on-demand projects. Use this query manually or once daily, not as a Grafana panel; INFORMATION_SCHEMA queries are not cached and can themselves incur a minimum processing charge.

Also use BigQuery dry runs before merging expensive scheduled queries. A dry run validates SQL and estimates bytes processed without charging for the query execution.

Title

Add cost-optimized governance event and aggregate tables

Description

This PR introduces a curated and cost-optimized analytics layer for GitHub Governance Intelligence.

Instead of querying raw normalized audit events directly from Grafana, the platform will:
- classify selected governance-sensitive audit actions into a curated governance-events table
- create time-based aggregate tables for dashboard use
- retain only dashboard-safe columns in the curated layer
- use partitioning, clustering, and partition-filter requirements to reduce query scan volume
- keep raw audit-log evidence in the existing GCS archive and normalized BigQuery table

This reduces repeated BigQuery scans from Grafana while keeping governance reporting explainable and auditable.
My recommendation on priorities

Do these in order:

1. Add partition-filter protection.
2. Create curated governance-events table.
3. Create 15-minute and daily aggregate tables.
4. Switch Governance Overview dashboard to aggregates.
5. Add retention only after compliance confirms policy.
6. Measure real query spend after one week.
